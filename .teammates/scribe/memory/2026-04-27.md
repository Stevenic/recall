## Wiki Spec v0.3 — Shared Wikis Added

stevenic asked to extend the wiki with a group-shared wiki concept: every agent has a private wiki by default, but a group of agents should be able to maintain one or more shared wikis for knowledge sharing. Each shared wiki has its own Vectra index, and at retrieval time the agent's search federates across the private index plus every shared index it is a member of.

Bumped `specs/wiki.md` from v0.2 → v0.3. Inserted a new §13 Shared Wikis (10 subsections) and renumbered §13–§21 → §14–§22. The cross-reference at the resolved Q5 entry (which previously pointed to "§13 and §15") was updated to "§14 and §16".

**New section §13 Shared Wikis covers:**
- §13.1 Motivation — glossary entities, cross-cutting decisions, themes spanning agents
- §13.2 Scope — synthesis-only; raw logs stay private; own root, own index, own slug namespace
- §13.3 Search federation — embed once, search every enabled index in parallel, apply per-index boost, merge, rerank; `source: { root, path }` on every hit
- §13.4 Authorship — three write paths (direct stub, promote from private, dreaming synthesis); v1 implements paths 1+2 only; sources support `private:<agent>/...` and `shared:<name>/...` URI prefixes
- §13.5 Slug resolution — bare `[[slug]]` resolves within same wiki; qualified `[[name:slug]]` resolves to a specific shared wiki; `private` is reserved and unaddressable from shared pages
- §13.6 Configuration — `wiki.shared[]` array of `{ name, path, role: member|reader, scoreBoost? }`; reserved name `private` rejected
- §13.7 Lifecycle — `recall wiki promote` copies private → shared and writes redirect stub at private location; demote not provided; cross-wiki rename/merge deferred to v2
- §13.8 Lint — `--include-shared` flag scans shared wikis; reports cross-wiki broken links
- §13.9 Disabled state — shared wikis opt-in; misconfigured paths reported at startup, excluded for the session, no impact on other targets
- §13.10 Anti-duplication — added a fourth surface-area question ("who is this knowledge for?"); rules: no private duplicate of shared page, no pre-sharing of stubs, sources may cross roots

## Wiki Spec v0.4 — Identity (Role Only)

**Context:** Beacon (during the previous turn that timed out) proposed an elaborate four-section `IDENTITY.md` design — Role + "What to summarize in daily logs" + "What deserves a wiki entry" + "What belongs in WISDOM" — to thread through all synthesis prompts. stevenic clarified: **identity is just the role.** "An engineer, a scientist, accountant, etc." The LLM is smart enough to derive what's worth summarizing/wiki-ing/wisdom-ing from the role itself; no need to pre-tune the prompt with explicit filtering rules.

**Action:** Bumped `specs/wiki.md` from v0.3 → v0.4. Inserted a new §11 Identity section between §10 Dreaming Integration and former §11 WISDOM.md Refactor. Renumbered §11–§22 → §12–§23.

**§11 Identity contents (6 subsections):**
- §11.1 Definition — identity is one or two sentences describing the role; explicitly NOT a list of summarize-rules / wiki-rules / wisdom-rules / persona narrative
- §11.2 File Format and Location — `<memory-root>/IDENTITY.md`, plain markdown, H1 with display name + a short paragraph
- §11.3 How Identity Threads into Synthesis — leading `<IDENTITY>` block in every synthesis prompt (compaction, wiki stub, dreaming, WISDOM); the synthesis prompts themselves do not restate filtering rules
- §11.4 Configuration — `identity.path` and `identity.name` in `.recall.yaml`; cached at session start, re-read on mtime change
- §11.5 Missing or Empty Identity — falls back to "a knowledge-worker agent" with a startup warning
- §11.6 Identity and Shared Wikis — identity is per-agent, not per-wiki; cross-agent shared-wiki contributions reflect each writer's lens

**Renumbering side effects:**
- 12 main section headings shifted (§11 → §12 ... §22 → §23)
- 22 subsection headings shifted (§11.x, §12.x, §13.x, §14.x, §15.x families)
- All in-text cross-references shifted via 12 sequential replace_all ops, highest-first to avoid double-shift conflicts
- Stale lint cross-reference (`recall wiki lint (see §11)` → after auto-shift would have been `(see §12)` which is WISDOM Refactor — wrong) corrected to `(see §15)` pointing at the WikiEngine API where the lint method is specified
- Changelog v0.2 and v0.3 entries restored to their original-version numbering (with parenthetical notes); v0.4 entry added describing the Identity insertion

**Decision rationale (saved for future me):**
The reason stevenic pushed back on beacon's elaborate IDENTITY.md is the same principle from §16 of WISDOM.md: "Simplify the model before shipping." A role-only IDENTITY.md has one knob (the role string). Beacon's proposal had four knobs (role + three filter lists), each of which would drift over time and require maintenance whenever a new synthesis pass is added. With role-only, adding a new synthesis pass just appends to the prompt template; no IDENTITY.md edits needed.

The trust here is in the LLM: a role like "litigation attorney" carries enough latent knowledge that the model can correctly score "client confidentiality concern" as wiki-worthy without being told. An explicit "wiki = client matters, not personal" rule would be redundant and brittle.
