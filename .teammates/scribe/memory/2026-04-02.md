## Memory Service Design Spec

Created `specs/memory-service.md` ‚Äî full design spec for the recall agent memory service.

**Key decisions:**
- Four pluggable abstractions: Storage, Embeddings, Index, Model
- Re-export Vectra's `FileStorage` and `EmbeddingsModel` interfaces (don't reinvent)
- New `MemoryIndex` interface to abstract over Vectra (allows SQLite swap later)
- New `MemoryModel` interface for LLM-powered compaction
- Default model adapter (`CliAgentModel`) spawns a CLI coding agent ‚Äî no API keys needed
- Default embeddings: transformers.js (`Xenova/all-MiniLM-L6-v2`) ‚Äî local, no cloud
- Compaction pipeline: daily‚Üíweekly‚Üímonthly‚Üíwisdom, with typed memory extraction
- Monorepo layout: `packages/core` + future plugin packages
- CLI name: `recall` (not `teammates-recall`)

**Open questions logged in spec:** compaction prompt tuning, auto-detect CLI agent, index isolation model, watch mode scope, monorepo vs single package.

**Files changed:** `specs/memory-service.md` (created)

## Spec v0.3 ‚Äî Feedback Round

Applied three feedback items from stevenic:

1. **`MemoryModel.complete()` return type enriched** ‚Äî Now returns `CompletionResult` with `text`, optional `inputTokens`/`outputTokens`, and `CompletionError` (code, message, retryable, retryAfterMs). Callers can detect rate limits and track token usage.
2. **File layout simplified** ‚Äî Replaced `.indexes/<name>/` (multiple named indexes) with a single `.index/` per memory root. Multiple agents/projects = multiple memory roots. Removed `MemoryIndexProvider` interface and `--index` CLI flag. `MemoryServiceConfig` now takes an optional `index: MemoryIndex` instead of `indexProvider` + `indexName`.
3. **Language bindings added** ‚Äî New `bindings/` folder with thin CLI wrappers for Python, Go, Rust, and C#. Each spawns the `recall` CLI subprocess and parses `--json` output. Python/Go in v0.2, Rust/C# in v0.3. New ¬ß9 in the spec covers design principles and binding surface.
