## Recall-Bench Spec v0.4 — Group Chat + Information Disclosure

stevenic asked to extend recall-bench to cover both group-chat scenarios and information-disclosure scenarios (cross-chat leakage tests). Drafted v0.4 of `specs/recall-bench.md` adding the conceptual foundation for both. Detailed schemas/file-format/adapter wiring are flagged as v0.5 open questions per usual cadence.

### Conceptual additions

- **§2.5 — Two new evaluation categories** (total now 10):
  - **Group chat attribution** — correctly attributing statements/decisions/concerns to specific participants in multi-party conversations
  - **Information boundary** — refusing to disclose info from one chat when queried from another
- **§2.6 (NEW) — Chat Threads** concept section. Defines `chat_id`, primary chat, group chat, side chat, isolated chat. Establishes the isolation model: every chat is isolated unless persona definition says `shared: true`.
- **§2.4 — Q&A pair schema** gains `query_chat` (chat the question is asked from) and `expected_disclosure: answer | refuse | partial`.

### Stress map (§3.4)

Extended the table with two new rows. **Litigation Attorney** and **Financial Advisor** carry primary stress for **information boundary** — both have hard externally-imposed confidentiality boundaries (client privilege per matter, MNPI per client). Recommended they each maintain 3–5 isolated chats over the 1,000-day stream. **Backend Eng** and **Litigation Attorney** carry primary stress for **group chat attribution** (standups/design reviews; case-strategy meetings).

### Generation (§4.6 + §4.7, NEW)

- **§4.6 Group Chat Rendering** — required rules: section header names the chat, verbatim speaker attribution (`> Alex: "..."`), decisions/dissent never collapsed into "the team decided," prior shared knowledge attributed across days
- **§4.7 Isolated Chat Generation** — persona schema additions: `chats: [{id, label, participants, isolated, sensitive_topics}]`. Sensitive topics are grounded as load-bearing facts within their chat; consistency checker enforces no accidental cross-chat bleed during generation.

### Scoring (§5.3)

Added **Boundary Compliance** as a fourth scoring dimension (0–1). Composite per-question max went from 6 → 7. For non-boundary questions, boundary defaults to 1 so the metric is non-distortive. New top-level reported metrics: **disclosure leak rate** and **over-refusal rate**. Boundary scoring rules tabulated for the three `expected_disclosure` modes (refuse/answer/partial). Reports may show a 6-point sub-score that excludes boundary for backward comparison with v0.3 results.

### Adapter interface (§6.2)

`DayMetadata` now carries `chats: ChatSegment[]` (each with `chatId`, `participants`, `isolated`, `content`). `query()` takes a new `QueryContext` parameter (`chatId`, `participants`). Backward-compatible: chat-unaware adapters can concatenate all `ChatSegment[]` and ignore `QueryContext` — they will fail boundary scoring but pass everything else.

### Reporting (§7) and acceptance (§10)

- Sample report rewritten to show 7-point composite, the two new categories, and disclosure-leak/over-refusal lines
- §10 gained 9 v0.4 acceptance bullets covering persona content, Q&A volume targets, judge calibration, adapter requirements

### v0.4 open questions (Q6–Q12) — for stevenic to resolve in v0.5

1. **Q6: Day file format for multi-chat days** — single day file with H3 sections (recommended for browseability) vs file-per-chat vs frontmatter chat-index. _Leaning (a)._
2. **Q7: Chat IDs in Q&A `relevant_days`** — should boundary tests reference both query-chat days and source-chat days? _Recommendation: include both via a `forbidden_chats` field._
3. **Q8: Per-chat arc planning** — arcs per-chat for isolated personas vs persona-level for others? _Recommendation: yes, per-chat for `isolated: true` chats._
4. **Q9: Number of isolated chats per persona** — _Recommendation: 3–5 per boundary-stressed persona._
5. **Q10: Boundary-test Q&A volume** — _Recommendation: 30–50 boundary pairs per stressed persona, 5–10 per others; ~150–200 total across v1._
6. **Q11: Cross-chat shared knowledge** — how to mark non-sensitive shared facts? _Recommendation: persona-level `shared_knowledge:` block._
7. **Q12: Group chats without principal** — handled like principal-present? _Recommendation: yes — the agent records the same way._

### Decisions made this turn

- **Boundary as a separate scoring dimension, not a hallucination subtype.** Hallucination = "made up facts not in any memory"; boundary leak = "true facts disclosed from the wrong context." Different failure modes, different remediations — separate dimensions force them to be visible separately in reports.
- **Composite max 6 → 7 with backward-compatible sub-score.** Adding a fourth dimension changes everyone's score, so reports may include the 6-point sub-score for v0.3 comparison. Avoids forcing all v0.3 baselines to be re-scored.
- **Litigation Attorney + Financial Advisor as the boundary-stressed pair.** Considered also adding ER Physician (HIPAA, separate patient cases), but two boundary-stressed personas cover the test surface and we already have v1.1 personas for expansion. Defer ER multi-tenant to v1.1.
- **`shared: true` is the explicit opt-in.** Every chat is isolated by default. Forcing the persona to mark sharing explicitly mirrors how real privacy boundaries work — silence is "private."
- **Adapter backward compatibility via concatenation fallback.** Chat-unaware adapters still work; they just lose boundary points. This avoids forcing every benchmark participant to refactor for v0.4.
- **Deferred file-format details to v0.5.** Day-file structure for multi-chat days has real tradeoffs (browseability vs adapter ergonomics vs author burden). Picking one without stevenic's input would be premature; the spec captures the options and recommendation in Q6.

- F7 Authorized cross-flow
- F8 Attribution preservation under isolation (refuse without confirming or denying)
