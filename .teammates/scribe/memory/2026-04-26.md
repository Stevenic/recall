## LLM Wiki Concept — Architectural Fit Review

stevenic shared Karpathy's "LLM Wiki" pattern and asked how to merge it into Recall. Per-memory-system wiki of facts, knowledge graph via cross-references, vectra-indexed with priority boost over time-based logs, WISDOM.md shrinks to entry points.

**My recommendation:** strong fit; the wiki is the synthesis layer Recall has been missing. Key shaping decisions I proposed:

1. **Wiki replaces dreaming's "insights" output** rather than living alongside it. Dreaming pipeline's primary job becomes wiki page create/update. `memory/dreams/insights/` and `memory/dreams/contradictions/` go away in favor of wiki edits with frontmatter flags.
2. **Vectra integration with configurable score multiplier** (~1.2–1.5×) rather than hard reordering — keeps recall predictable when query is genuinely temporal.
3. **WISDOM.md becomes principles-only** plus a curated "Knowledge Map" section linking to high-traffic wiki pages. Topical entries that crept into WISDOM over time migrate to wiki.
4. **Pointer-expansion is free** — wiki cross-references are exactly the hierarchical memory pattern; no new mechanism needed.

**Tradeoffs flagged:**
- Surface area expansion (raw log + typed memory + wiki = same fact in three places if rule isn't crisp). Proposed rule: raw logs = immutable history, typed memories = atomic durable facts, wiki = topical synthesis.
- Dreaming quality bar gets much higher — sloppy synthesis corrupts the wiki rather than producing ignorable insights.
- Regeneration story matters — wiki must be regenerable from raw logs (matches "summaries never replace source data"). Implies `recall wiki rebuild` command and `sources: [...]` frontmatter on every page.
- Per-memory-system scope is correct — each agent's memory root gets its own wiki; team-shared knowledge stays in `docs/`.

**Next step proposed:** draft wiki spec v0.1 covering page format, naming, link syntax, index.md, dreaming integration, vectra boost mechanics, regeneration command, and the WISDOM-vs-wiki split rule. Also revise `specs/dreaming.md` to make wiki edits the canonical output.

Awaiting user direction before drafting. No files modified this turn — exploratory architecture discussion only.

## Back-Pointers — Generalization Insight

stevenic surfaced what they called the "real breakthrough" under Karpathy's wiki idea: it's not the wiki, it's the **pattern of memories citing other memories that the agent can chase on demand**. Their concrete example: today's daily log can write a `[[2026-04-12]]` back-pointer when the agent recalls something it did two weeks ago. When today's log is recalled in the future, the agent can chase the pointer.

**My analysis (no spec drafted yet, exploratory):**

This is a generalization of three existing pointer mechanisms in Recall:
1. Structural pointers (parent→child) in `specs/hierarchical-memory.md`
2. Wiki content pointers (`[[slug]]`) in `specs/wiki.md`
3. (NEW, proposed) Associative pointers in any memory file pointing to any other memory

The unifying insight: **all three should be the same `[[ref]]` syntax with a namespace-aware resolver**. Daily logs use date slugs (`[[2026-04-12]]`); typed memories use file stems (`[[feedback_testing]]`); wiki uses topic slugs (`[[auth-middleware]]`); weekly summaries use ISO weeks (`[[2026-W15]]`). Pointer-chasing already exists from the hierarchical-memory spec — this just adds more lookup namespaces.

**Why this is more powerful than the wiki alone:** wikis require synthesis (someone decides a topic deserves a page). Daily-log back-pointers require only that the agent recognized a connection while writing — a much lower bar. The two complement each other:
- Daily-log back-pointers = associative trails (Bush's Memex pattern)
- Wiki pages = distilled topics
- Hierarchical pointers = structural rollup

**Implication for WISDOM.md shrinking:** I'd argued WISDOM shrinks because the wiki replaces topical content. With back-pointers, the case is stronger — most cross-temporal threads can live as inline `[[ref]]` in daily logs themselves, never needing to be promoted to a wiki page at all.

**Shaping decisions I asked stevenic to resolve:**

1. Who writes back-pointers? Agent-in-real-time (high signal, sparse) vs. compaction/dreaming post-hoc (uniform but lower judgment) vs. both. I lean both with agent-real-time as primary.
2. Slug namespace dispatch — confirm shapes are non-conflicting (date vs. ISO week vs. snake_case typed vs. kebab-case wiki).
3. Pointer rot under compaction — daily files are eidetic so links survive; wiki has redirect-stub story; need to confirm typed memory rename story.
4. Backlinks (forward in time, auto-generated) — defer.
5. Where to spec — three options I floated:
   - **A**: fold into `specs/hierarchical-memory.md` as a "Memory Links" section
   - **B**: standalone `specs/memory-links.md` v0.1, foundational, referenced by wiki + hierarchical-memory + dreaming
   - **C**: defer; absorb into wiki v0.2 as a generalized link section

**My recommendation:** Option B. Foundational enough to deserve its own spec; three dependent specs already need the concept; lets us answer "who writes them" once instead of three times.

**Next turn:** stevenic to confirm option (A/B/C) and the agent-real-time write model. If confirmed, I draft `specs/memory-links.md` v0.1 covering syntax, resolver namespaces, write conventions, chasing budget, and integration contracts back into wiki + hierarchical-memory + dreaming specs.

## Typed Memories vs Wiki — Recommend Collapse

stevenic asked: "do we need the typed memories if we have the wiki?"

**My answer:** No — the wiki can subsume them. Recommended collapsing the tier from 3 → 2 (raw / wiki).

**Mapping:** typed-memory categories map cleanly to wiki categories:
- `user` → entity
- `feedback` → concept
- `project` → project
- `reference` → reference

A typed memory becomes a wiki *stub* — same frontmatter category, single source (`agent-observation`), short body, single paragraph. Dreaming enriches stubs into multi-source pages over time. Same directory (`memory/wiki/`), same `[[slug]]` resolution. Stub vs rich is just `len(sources)`.

**What we gain:**
- Surface-area rule (§12) drops to two tiers — eliminates the "is this typed or wiki?" judgment call
- Stubs evolve into rich pages naturally — no more parallel `feedback_testing.md` typed file and `auth-middleware.md` wiki file on overlapping topics
- WISDOM Knowledge Map points at one tier, not two
- `specs/memory-links.md` namespace dispatch simplifies (no more snake_case typed namespace; just dates / ISO weeks / wiki slugs)

**What we lose:** nothing functional. Only the file path changes for agent-in-the-moment writes.

**Caveat — discipline preservation:** the four typed-memory categories' write conventions (`feedback` → "rule + **Why:** + **How to apply:**", etc.) must carry over as **per-category page templates** in `specs/wiki.md` §3. Otherwise we lose the discipline that made typed memories useful.

**Spec implications I proposed:**
1. `specs/wiki.md` §12 — drop typed-memory column from surface-area rule
2. Wiki open question #5 ("min sources to create a page") resolves as **1 for agent stubs, 3 for dreaming-promoted synthesis**
3. `specs/wiki.md` §3 — add per-category page templates carrying typed-memory write conventions

**Migration cost flagged:** ~30 existing typed memories across teammates need to move to `memory/wiki/` with category preserved in frontmatter. The `<type>_<topic>.md` filename convention dies; slugs become `<topic>.md` only. WISDOM compaction logic shifts source from typed memories → wiki pages. One-time, scriptable, but not free.

**Awaiting:** stevenic decision on whether to fold this into `specs/wiki.md` v0.2 alongside the existing 8 open-question resolutions.
