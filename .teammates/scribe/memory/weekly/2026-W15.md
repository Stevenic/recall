## Hierarchical Memory Spec ‚Äî v0.1 Draft

Created `specs/hierarchical-memory.md` ‚Äî full design spec for hierarchical (eidetic) memory architecture.

**Key design decisions:**
- Raw memories are **never deleted** ‚Äî permanent source of truth (~5 MB for 3 years)
- Parent nodes (weekly/monthly) store `pointers` array in YAML frontmatter + generated summary
- Two embeddings per parent: aggregated child embedding (`#agg`) and summary embedding (`#summary`)
- Aggregated embedding = normalized mean of child embeddings (uniform weighting for MVP)
- Two-phase recall: Phase 1 (high recall ‚Äî parent vector + raw vector + BM25 in parallel), Phase 1b (expand pointers recursively), Phase 2 (rerank with hybrid scoring: embed + BM25 + parent boost)
- Phase 2 scoring: `w_embed=0.5, w_bm25=0.3, w_parent=0.2` (configurable)
- Parent oversampling (default K=10) to compensate for aggregation washout
- Summaries included in results as separate type (gap coverage, not authoritative)

**New interfaces:**
- `MemoryIndex.getEmbedding()` and `upsertEmbedding()` for aggregated embedding support
- Extended `SearchResult` with `isSummary`, `parentUri`, `scoreBreakdown`
- Extended `QueryOptions` with `parentCandidates`, `rawCandidates`, `scoringWeights`, `enableBM25`
- New `HierarchicalMemoryConfig` for all tuning knobs
- Removed `dailyRetentionDays` and `weeklyRetentionWeeks` from `CompactionConfig`

**Implementation sequencing:** 5 phases (A‚ÄìE), A+B parallel, D parallel with C, E anytime after A.

**5 open questions logged** ‚Äî BM25 implementation choice, salience weighting signals, max expansion depth, summary result type, cross-hierarchy pointer strategy.

**Files created:** `specs/hierarchical-memory.md`

## Hierarchical Memory Spec ‚Äî v0.2 (Open Questions Resolved)

Updated `specs/hierarchical-memory.md` to v0.2 per stevenic's decisions on all 5 open questions.

**Decisions applied:**
1. **BM25** ‚Äî Use Vectra's built-in BM25 support, no additional library
2. **Salience weighting** ‚Äî Defined 3-signal model (token count 0.4, entity density 0.3, decision markers 0.3); deferred to v2, MVP uses uniform
3. **Max expansion depth** ‚Äî Always expand to leaf, no depth cap. Added **temporal affinity** scoring: multiplicative date-proximity boost when query contains time references, no recency decay so old memories are never penalized
4. **Result type** ‚Äî `ResultType` enum (`RAW` / `SUMMARY`) replaces `isSummary` boolean
5. **Cross-hierarchy pointers** ‚Äî Group/team nodes point directly to raw memories

**New spec content:**
- Temporal affinity formula: `exp(-|date_i - date_ref| / œÉ)` with configurable œÉ (default 30 days)
- Temporal affinity as multiplier on final score (neutral when no time reference detected)
- `temporalSigma`, `temporalAffinity` added to `HierarchicalMemoryConfig`
- `temporalReference` added to `QueryOptions`
- New acceptance criteria for temporal affinity behavior
- 2 new open questions: temporal reference extraction sophistication, salience signal extraction timing

## Dreaming System Spec v0.1

Created `specs/dreaming.md` ‚Äî full design spec for Recall's asynchronous knowledge synthesis system (dreaming). Complements the existing compaction pipeline by adding cross-temporal analysis capabilities.

**Key design decisions:**
- Three sequential phases: Gather (signal collection) ‚Üí Analyze (LLM synthesis) ‚Üí Write (persist outputs)
- Signal-driven candidate selection: search logs, entity frequency, typed memory staleness, wisdom drift
- Five candidate scoring signals with configurable weights: hit frequency (0.25), query diversity (0.25), gap signal (0.20), staleness (0.15), entity frequency (0.15)
- Five analysis templates: cross-reference, gap analysis, contradiction detection, typed memory extraction, theme synthesis
- Three output types: insight files (`memory/dreams/insights/`), contradiction flags (`memory/dreams/contradictions/`), typed memory promotions
- Append-only dream diary (`DREAMS.md`) for observability
- Machine state in `.dreams/` (gitignored): search logs, candidates carry-over, dream state
- Opt-in via `DreamingConfig.enabled`, scheduled via cron or on-demand `recall dream`
- Eidetic-compatible: never modifies or deletes existing memory files
- 5 open questions logged for v0.2 resolution

**Files created:** `specs/dreaming.md`
