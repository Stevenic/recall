## 2026-04-28

### `recall-bench generate` — improved console log format

User asked for the generate tool to (1) report the pass being worked on (arc vs gap) when logging to console, and (2) stop overwriting previous progress lines — stream them sequentially. Scribe drafted a mockup with `[arc]/[gap]/[anchor]` labels, but reading the generator I confirmed the only two passes are `runArcPass` and `runGapPass` (no `anchor`). So the kind is `'arc' | 'gap'`.

**Changes:**

1. **`packages/recall-bench/src/generator-types.ts`** — extended `GeneratorConfig.onDay` from `(dayNumber, content) => ...` to `(dayNumber, content, kind: GeneratedDayKind) => ...` and exported new `GeneratedDayKind = 'arc' | 'gap'` type.

2. **`packages/recall-bench/src/generator.ts`** — `runArcPass` now invokes `onDay(..., 'arc')`; `runGapPass` invokes `onDay(..., 'gap')`. No other behavior change; storage and recent-day insertion paths untouched.

3. **`packages/recall-bench/src/cli.ts`** — `generate` command's `onDay` now uses `console.log` (newline) instead of `process.stdout.write('\r ...')`. Each line is `  [arc]  day  N/end  day-NNNN.md  (M unique)`. The post-loop `console.log('')` newline-after-progress shim is removed (no longer needed). `--json` mode is unchanged: still emits machine-parseable summary only.

4. **`packages/recall-bench/tests/generator.test.ts`** — added a new test `passes the generation kind (arc vs gap) to onDay` that asserts both kinds are observed when arcs cover only part of the window. Existing tests that didn't read the kind argument still pass (extra arg is just ignored).

**Pre-existing CLI bug noticed (not fixed):** `cli.ts:336` does `await import(resolve(modulePath))` for `--model <path>`. On Windows, `resolve()` produces `C:\...\file.mjs`, but Node ESM's loader requires `file://` URLs for absolute paths. This means `--model /path/to/stub.mjs` already fails on Windows with `ERR_UNSUPPORTED_ESM_URL_SCHEME`. Not in scope for this task; flagging here so future work can fix with `pathToFileURL()`. Same issue likely affects `--adapter <path>` and `--judge <path>`.
